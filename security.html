Security

Last updated: 2026-02-15

Setfeed is local-first and end-to-end encrypted. The web bridge (setfeed.app) is a ciphertext transport layer: it enforces policy and delivers encrypted messages without access to message content.

End-to-end encrypted
Ciphertext-only storage
Device-held private keys
Rolling is constrained by design

Encryption boundary

Messages sent through setfeed.app are encrypted in the browser to the recipient’s public key. Only the recipient device holds the private key required to decrypt.

What we store

  • Encrypted message ciphertext
  • Key version and delivery metadata (timestamps, routing)
  • Abuse-prevention counters and hashed operational identifiers

What we do not store

  • Message plaintext
  • Recipient private keys
  • Content-derived analytics

If a design change requires server-side plaintext, we treat it as a bug.

Web bridge flow

Browser encrypts → Cloud Functions validate policy → Ciphertext stored → Push ping → App fetches + decrypts

Receive codes and sending rules

Setfeed supports three receive code types. Two are signed; one is capability-based and strictly limited.

Secure and Personal (signed)

  • Secure and Personal sends require email-link sign-in.
  • Rate limits apply to reduce abuse and protect recipient devices.

Rolling (capability-based)

  • Rolling sends do not require sign-in.
  • They are short-lived, rate-limited, and typically single-use.
  • App Check is required to reduce automated abuse.

Abuse prevention (high level)

We enforce strict controls to keep Setfeed usable without becoming an open inbox. Controls include short validity windows, usage limits, payload size caps, App Check, and rate limiting.

We intentionally do not publish exact thresholds on this page. Guardrails may change as we observe real-world abuse.

Responsible disclosure

If you believe you’ve found a security vulnerability, we appreciate responsible reporting.

Report by email

security@setfeed.app

Please include

  • A clear description of the issue
  • Steps to reproduce
  • Potential impact
  • Proof-of-concept (if available)

Guidelines

  • Do not access or modify other users’ data.
  • Do not perform denial-of-service tests.
  • Share only what’s necessary to demonstrate the issue.

We aim to acknowledge reports within 72 hours and coordinate a disclosure timeline.

FAQ

Can Setfeed read my messages?

No. Messages are encrypted before they reach our servers. We store ciphertext only.

What happens if the server is breached?

An attacker could access encrypted ciphertext. Without device private keys, messages cannot be decrypted.

Is Setfeed a chat app?

No. Setfeed is intentionally one-way, scheduled, and local-first.
